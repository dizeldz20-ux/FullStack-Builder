# Voice-agent runtime hardening pattern

Use this reference when hardening a browser voice app that routes microphone/STT output through a backend agent bridge and paid provider APIs.

## Durable invariants

- Browser STT provider keys stay backend-only. The browser may receive a short-lived provider token, never the provider API key.
- Expensive runtime endpoints should have a lightweight app-level gate when exposed through tunnels or public previews:
  - optional shared bearer token for dev/preview (`RUBY_VOICE_API_TOKEN`-style backend var + `VITE_*` browser var)
  - fail closed in production if the gate is required but not configured
  - in-memory per-bucket/IP/origin rate limit for local/simple deployments
  - do not block `OPTIONS` preflight
- Keep health/version endpoints readable without the voice token, but make version metadata safe:
  - service/version/mode/runtime/provider-configured booleans are OK
  - no cwd, filesystem paths, full env, raw secrets, or API keys
- If backend invokes an agent/LLM CLI per turn:
  - never pass user speech/prompt text in argv
  - send the prompt via stdin or another non-argv channel
  - do not pass `process.env` wholesale to the child
  - use a minimal allowlist and exclude provider/app secrets the child does not need
  - propagate HTTP request aborts into the child process and terminate it

## Test pattern

Add regression tests for:

1. Safe version endpoint: returns expected metadata and does not contain secret/env/path strings.
2. Child-process bridge:
   - prompt arrives via stdin
   - argv does not include user text
   - child env excludes provider/app secrets
   - abort terminates the child process
3. Protected endpoints:
   - unauthenticated request returns 401 when token is configured
   - authenticated request reaches route behavior
   - health/version still work without the token
4. Frontend request helpers:
   - `Authorization: Bearer ...` is sent when the public preview token env var is configured
   - provider API keys are not referenced in frontend source

## Runtime smoke pattern

After tests/build, start the backend on a confirmed-free high port with mock mode and a dummy token. Verify:

- `/api/health` returns 200 without auth
- `/api/version` returns 200 and safe metadata without auth
- protected route without bearer returns 401
- protected route with bearer returns 200/expected route behavior
- STT session-token endpoint without bearer returns 401 when the app-level gate is enabled

Before trusting a smoke failure, inspect response headers/body. A 401 with unrelated `server: ttyd` or Basic realm can mean the port belonged to another service, not the app. Pick a different confirmed-free port and rerun.

## Commit hygiene

- Remove runtime logs generated by smoke tests before committing.
- Run `git diff --check` and a secret-like diff scan; treat matches as prompts for inspection, not automatic failures when they are env variable names or fake test values.
- Commit after full tests/typecheck/build and runtime smoke are green.
