# PRD Quality Checklist

Validates a generated PRD against the `prd-generator` skill's quality bar. Run after `@tasks/generate-prd.md` step 6 (self-audit) and again when user runs `/prd-generator check <path>`.

## Purpose
Ensure every PRD is handoff-ready for `build-product` and `writing-plans` — no missing sections, no invented features, no vague success metrics, no unprioritized open questions.

---

## 1. Structure (the file actually has the right sections)

- [ ] File lives at `plans/PRD-<slug>.md` (kebab-case, max 4 words)
- [ ] Header has: Slug, Date, Owner, Status, Version
- [ ] TL;DR section present
- [ ] Problem Statement section present
- [ ] Target User section present
- [ ] Goals & Non-Goals section present
- [ ] Core Feature (MVP) section present
- [ ] User Flow section present (table format)
- [ ] Edge Cases & Failure Modes section present (table format)
- [ ] Acceptance Criteria section present (5-15 items)
- [ ] Success Metrics section present
- [ ] Out of Scope section present (2+ items)
- [ ] Technical Constraints section present (or "To be decided by `build-product`")
- [ ] Assumptions section present
- [ ] Open Questions section present
- [ ] Handoff Notes section present (to `build-product` + to `writing-plans`)

**If any section is missing → FAIL.** Even an empty section needs the heading + "Not specified — see OQ-N".

## 2. Specificity (the content is concrete, not vague)

### Target User
- [ ] Persona has a name (not "User 1")
- [ ] Age range OR role is specific
- [ ] Tech comfort level stated
- [ ] Current tool / workaround mentioned
- [ ] "Why they'll switch" is concrete (not "it'll be better")

### Core Feature
- [ ] ONE feature, not a list
- [ ] Sub-capabilities (if any) are clearly part of same MVP
- [ ] "Why this is the core" justified

### User Flow
- [ ] 2-7 steps (not 1, not 10+)
- [ ] Table format (Step / User Action / System Response / Notes)
- [ ] Entry point stated
- [ ] Exit / success state stated

### Edge Cases
- [ ] At least 3 categories covered: Network, Empty State, Invalid Input
- [ ] Auth-related edge case (if auth applies)
- [ ] Each has Priority (P0/P1/P2)

### Success Metrics
- [ ] At least 1 metric with a number
- [ ] Timeframe specified (days / weeks)
- [ ] "How to measure" is concrete (analytics event, query, manual check)

## 3. Scope Discipline (the Out-of-Scope actually exists)

- [ ] Out of Scope has 2+ items
- [ ] Each item explains "why not now" (not just "later")
- [ ] Out of Scope items are not contradicted by features elsewhere in the PRD

## 4. Open Questions Hygiene

- [ ] Open Questions enumerated OQ-1, OQ-2, ...
- [ ] Each OQ has: Context, Blocks, Suggested resolution, Priority
- [ ] P0 OQs are flagged explicitly
- [ ] If section is empty, says "None — all questions resolved during interview"
- [ ] Cross-reference to @frameworks/open-questions-catalog.md when applicable

## 5. Anti-Invention (nothing was made up)

- [ ] Every feature in Core Feature / Sub-capabilities traces to interview Q4 or explicit user statement
- [ ] No login / auth if user didn't mention it
- [ ] No notifications if user didn't mention it
- [ ] No "share with friends" / social if user didn't mention it
- [ ] No analytics / tracking if user didn't mention it
- [ ] Assumptions section flags every inferred item with `[inferred]`

**If you find an invented feature → REMOVE IT and move to Open Questions if needed.**

## 6. Handoff Readiness

- [ ] "To `build-product`" notes mention: start with Core Feature, respect Out of Scope, P0 edge cases
- [ ] "To `writing-plans`" notes mention: Acceptance Criteria basis, User Flow ordering, Technical Constraints
- [ ] A P0 Open Question (if any) is called out as "MUST RESOLVE BEFORE BUILD"

## 7. Language & Format

- [ ] Body in Hebrew (or user's language)
- [ ] Tech terms in English (API, endpoint, MVP, schema, etc.)
- [ ] No walls of prose — tables and bullets where applicable
- [ ] No marketing fluff ("amazing", "revolutionary", "next-gen")
- [ ] Code blocks (if any) use fenced markdown with language hint

## 8. Provenance

- [ ] Footer: "PRD generated by prd-generator skill · Built with Skillsmith · Skillsmith"
- [ ] Generated date matches the Date field in header

---

## Scoring

| Score | Status | Action |
|-------|--------|--------|
| All checks pass | ✅ Handoff-Ready | Send to `build-product` / `writing-plans` |
| 1-3 fail | ⚠️ Minor Issues | Fix inline, then handoff |
| 4+ fail | ❌ Not Ready | Re-run interview for the missing pieces, regenerate |
| Any P0 OQ unresolved + not flagged | 🚨 Blocker | Resolve before any build starts |

## When to Run This Checklist

- After every PRD generation (self-audit in `@tasks/generate-prd.md` step 6)
- When user runs `/prd-generator check <path>`
- Before handing off to `build-product` or `writing-plans`
- Before publishing the PRD to a repo or to a team
