# Product Build Blueprint — 7 השלבים

> גרסה קלה ופשוטה של build-product. מתאים לפרויקטים קטנים, סקריפטים, בוטים, אוטומציות. לא מתאים למוצרים אמיתיים (לאלה יש build-product).

## העיקרון העליון

> לפני שכותבים קוד — מבינים. לפני שמתכנתים — מתכננים. לפני שמעלים — בודקים.

---

## שלב 1: הבנת הצורך

### 5 שאלות חובה

1. **מה בדיוק צריך להיבנות?** (2-3 משפטים)
2. **למי זה מיועד?** (משתמש סופי / צד שלישי / פנימי)
3. **איפה זה ירוץ?** (שרת / דפדפן / דסקטופ / נייד / וואטסאפ)
4. **מה הקריטריונים להצלחה?** (בדיקה אחת שאפשר להריץ ולראות שזה עובד)
5. **מה בהיקף ומה לא?** (scope / non-scope)

### תבנית הבנה מהירה (למלא ולהציג למשתמש)

```
## מה: [תיאור קצר]
## למי: [משתמש/לקוח/פנימי]
## איפה: [ווב/דסקטופ/API/WhatsApp]
## בהיקף: [מה נכנס]
## מחוץ להיקף: [מה לא נכנס]
## הצלחה = [איך נבדוק שזה עובד]
```

### אם הבקשה מעורפלת

אל תנחש — שאל. דפוסים מעורפלים:
- "אני רוצה משהו שעושה X" → מה זה אומר בדיוק?
- "תבנה לי בוט" → איזה בוט? לאיפה? מה הוא עושה?
- "זה דומה ל-Y" → אבל במה זה שונה?

## שלב 2: בחירת ה-Stack

### עץ החלטות (התחל מלמעלה)

```
האם צריך ממשק משתמש?
├── כן → React / Next.js / HTMX / Hyperframes
│         ├── דסקטופ/נייד → Capacitor / Tauri
│         └── ווב בלבד → Next.js / Vite
└── לא → Node.js / Python / Bash
         ├── אוטומציה/סקריפט → Python / Bash
         ├── API/שרת → Node.js / Python
         └── בוט (WhatsApp/Telegram) → Node + GreenAPI
```

### מתי לבחור מה

| צורך | Stack מועדף |
|---|---|
| ווב פרונטנד | React + Tailwind |
| API פשוט | Node.js + Express |
| אוטומציה/קלסרים | Python |
| בוט וואטסאפ | Node.js + GreenAPI |
| קונטיינר | Docker + Node/Python |
| שליפת מידע מרשת | Playwright / curl |
| TTS/קול | ElevenLabs |
| ממשק ווקלי | Hyperframes + WebRTC |

### ⚠️ מתי לא להשתמש בסקריפט הזה

אם התשובה לאחת מהשאלות הבאות "כן" → עבור ל-`build-product`:

- [ ] יש Supabase auth + RLS?
- [ ] צריך deploy ל-Cloudflare Workers/Pages עם Access?
- [ ] זה Next.js 15 עם App Router?
- [ ] המוצר צריך לשרוד כמה sessions (state tracking)?
- [ ] הולך להיות deploy אוטומטי בכל שינוי?

אם כל התשובות "לא" → `product-build-blueprint` זה הכלי הנכון.

## שלב 3: תכנון

### לפני שמקלידים קוד — 3 דברים

1. **מבנה התיקיות** — איפה כל דבר יחיה
2. **API/פורטים** — מה נחשף, מה פנימי
3. **נקודות כאב** — איפה דברים עלולים להישבר

### תבנית תכנון קצרה

```
## מבנה
project/
├── src/          # קוד ראשי
├── tests/        # בדיקות
├── scripts/      # סקריפטים
└── README.md     # התחלה מהירה

## API endpoints
GET  /health      → בדיקה
POST /ask         → הלוגיקה הראשית

## סיכונים
• [סיכון 1] → [מה עושים אם קורה]
```

### כללי תכנון

- **תיקיות שטוחות ככל האפשר** — לא over-nest
- **API יציב** — גם אם משתנה, ה-shape נשאר
- **נקודות כאב** — איפה הסיכון? (DB connection, third-party API rate limits, race conditions)

## שלב 4: בנייה

### סדר נכון (עקבי!)

```
1. סקפולדינג — תיקיות + קובץ ראשי + README
2. תסריט התחלה — שרת עולה, אפשר להריץ
3. לוגיקה בסיסית — הכי פשוט שעובד
4. אינטגרציות — חיבור לכלים חיצוניים
5. שגיאות — מה קורה כשמשהו נכשל
6. בדיקות — שרת עונה נכון לפחות למקרה אחד
7. ניקיון — למחוק debug, לתעד
```

### כללי קוד

- **פשוט קודם** — אל תייעל לפני שזה עובד
- **שם ברור** — `handleIncomingMessage` עדיף על `fn1`
- **שגיאות נכנסות** — try/catch עם הודעה ברורה
- **קונפיג בקובץ** — לא hardcoded בקוד
- **לא לחזור על קוד** — פונקציה אחת, לא שלוש

### שימוש בסקריפטים

```bash
# סקפולד Node.js
bash scripts/scaffold-node.sh my-project

# סקפולד Python
bash scripts/scaffold-python.sh my-project
```

## שלב 5: בדיקות

### מה לבדוק

| מה | איך |
|---|---|
| השרת עולה | `curl http://localhost:PORT/health` |
| הלוגיקה עובדת | בדיקה ידנית של המקרה המרכזי |
| שגיאות מטופלות | לשבור משהו בכוונה, לראות מה קורה |
| credentials | לוודא שאין secrets בקוד |

### בדיקה מינימלית לכל פרויקט

```bash
# 1. התקנה נקייה
cd project && npm install  # או pip install -r requirements.txt

# 2. התחלה
npm start  # או python3 main.py

# 3. בדיקה
curl http://localhost:PORT/health

# 4. המקרה המרכזי
# [פקודה ספציפית לפי הפרויקט]
```

### Checklist מלא

ראה `@frameworks/checklist.md` לרשימה מלאה לפני העלאה.

## שלב 6: העלאה / Deployment

### שאלות לפני העלאה

1. **איפה זה ירוץ?** (שרת ישיר / Docker / Cloudflare / Railway)
2. **איך מתחילים?** (`npm start` / `docker run`)
3. **מי מפעיל?** (the user / CI / cron)
4. **האם צריך domain?** (כן/לא)

### אפשרויות Deploy

| צורך | כלי |
|---|---|
| **מוצר web אמיתי** (יש לי auth + DB) | `cloudflare-deploy` skill (Next.js/Workers/Pages) |
| **סקריפט/בוט שרץ 24/7** | Docker + VPS (או Railway/Fly.io) |
| **סקריפט חד-פעמי** | `crontab` או GitHub Actions |
| **קובץ סטטי** | Pages או אפילו GitHub Pages |
| **לוקאלי בלבד** | אין deploy — רק `npm start` |

### Docker — מתי ואיך

**מתי:** כשצריך סביבה קבועה, או כשיש תלות מורכבת (Python + FFmpeg + Whisper).

**איך:** `docker-essentials` skill → רואים איך לבנות ולהריץ.

### לא לשכוח

- `README.md` עם התחלה מהירה
- `.env.example` עם כל המשתנים (בלי ערכים אמיתיים)
- `health` endpoint — תמיד

## שלב 7: תיעוד

### מה לתעד

```
README.md
├── מה זה עובד (2 שורות)
├── איך מתחילים (3-5 פקודות)
├── מה ה-API (אם יש)
└── איך להתקין מחדש
```

### מה לא לתעד

- היסטוריה של מה שיניתי
- הערות מיותרות בקוד (קוד ברור = לא צריך הערות)
- דברים שכבר יש בקוד
